# Ecosystem Integration Handoff — 2026-03-15

Context for the AI assistant working in the application pipeline. This document describes infrastructure changes made in the meta-organvm session that directly affect how this pipeline operates.

---

## What Changed

The ORGANVM system now describes itself through a chain of data exports that feed into this pipeline automatically. Previously, metrics were hardcoded in `config/metrics.yaml` and manually maintained. Now they are auto-generated from the living system.

### Data Flow Into This Pipeline

```
Engine pulse daemon (every 15min)
  → ontologia store (variables, metrics, observations)
  → Neon PostgreSQL (pulse_snapshots, metric_observations)
  → system-snapshot.json (narrative export, committed to corpus)

organvm refresh (daily CI at 6am UTC + manual)
  → system-metrics.json (computed metrics with workspace scan)
  → system-snapshot.json (enriched with code profile, word counts)
  → committed to corpus repo by CI

Pipeline consumption:
  → refresh_from_ecosystem.py reads system-snapshot.json
  → writes config/metrics.yaml (auto-generated, never edit manually)
  → check_metrics.py --fix propagates into blocks/resumes
```

**Command to refresh:** `make refresh-ecosystem` (or `make refresh-all` for full cycle)

- "documentation engineer" / "technical writer" → `documentation-engineer`
- "ai safety" / "trust and safety" / "compliance" → `governance-architect`
- "platform engineer" / "developer experience" / "devex" → `platform-orchestrator`
- "fractional cto" / "founding engineer" / "entrepreneur in residence" → `founder-operator`
- "curriculum" / "instructional design" / "edtech" → `educator`
- "creative technologist" / "generative ai" → `creative-technologist`
- Everything else → `independent-engineer` (default)

Run `python scripts/classify_position.py` standalone to see the full distribution and reclassification opportunities.

## Known Gaps & Limitations

1. **`automated_tests: 46`** in metrics.yaml is "repos with test suites," not test count. Use `verified_passing_tests: 3421` for the actual number. The field name is legacy — don't change it (other scripts depend on it).

2. **`total_words_numeric`** requires workspace access to compute. In CI (no workspace), this field comes from the persisted `system-metrics.json`. Run `calculate-metrics.py` locally after major content changes to update.

3. **Engagement data is small** — 10 engaged, 11 silence for the recalibration. The correlations are directional signals, not statistically robust. They'll improve as outcomes accumulate.

4. **Surfaces 9, 10, 11 are infrastructure-ready but data-thin:**
   - Surface 9 (scoring weights): `recalibrate_engagement.py` works but needs N=50+ outcomes with dimension scores for statistical confidence
   - Surface 10 (network intel): `github_proximity.py` scans GitHub events but ORGAN-VI/VII don't have CRM data yet — pipeline CRM is in `signals/contacts.yaml`
   - Surface 11 (block-outcome): `block_engagement.py` works but positive outcome signal (acceptances) is needed for golden/toxic classification. Currently using read-vs-silence as proxy.

5. **New resumes (positions 6-9)** exist as HTML base templates. They have NOT been through the batch-tailoring process. Before submitting with a new position, create a tailored version in `materials/resumes/batch-04/`.

6. **Framing blocks for new positions** are single-tier (`tier: single`). The identity blocks have 4 tiers (60s/2min/5min/cathedral) but framing blocks are inherently single-tier by convention. No action needed.

## Stakeholder Portal

The stakeholder portal is **LIVE** at `https://stakeholder-portal-ten.vercel.app/`. It auto-deploys on push to main. It now has:
- `/api/metrics` — queries Neon for live pulse snapshots + metric time-series
- `/dashboard` — live density, entities, edges, tensions + 7-day sparkline
- `/organs/[key]` — per-organ density + gate pass rate
- `/api/knowledge` — structured organ data from manifest
- `/ask` — AI chat with hybrid retrieval

The portal reads from the same Neon database that the pulse daemon writes to. The pulse daemon runs every 15 minutes on the local machine via LaunchAgent.
